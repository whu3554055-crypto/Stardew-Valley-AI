# AI 与 Agentic 开发指导大纲

> **性质**：方向性、契约性、可反复自查；与 `TODO.md`、托管迭代清单互补，不替代具体任务排期。  
> **用途**：后续开发的**优先级锚点**、阶段性反思、降低「传统碎规则」与「AI 层」互相打架导致的架构漂移。  
> **修订**：当 Agentic 边界或存档契约变更时更新本文，并在 `CHANGELOG` 或提交说明中留一句。

## 1. 我们要的「新范式」指什么（现实定义）

本项目期望 **AI / Agentic** 不仅让皮肤更「活」，更在开发上体现为：

| 维度 | 传统范式（仍需要，且要写好） | 新范式（本项目要刻意加强的） |
|------|------------------------------|--------------------------------|
| **世界与手感** | 确定性循环、物理、输入、UI、存档 | 不替代，但用**数据与契约**减少魔法数与分叉爆炸 |
| **叙事与 NPC** | 手写分支、固定台词 | **结构化生成 + 记忆 + 校验 + 降级**，模型填槽而非乱改状态 |
| **任务与经济** | 静态表 | **玩家行为 → 世界状态 →（可选）推理 → 校验 → 写回** 的可观测管道 |
| **开发流程** | 一切写死在脚本 | **Schema、表、工具链、日志**；人能审、机能验 |

**结论**：「新范式」≠ 删掉所有 `if`；**= 在明确边界内，用 AI 与 Agent 承担「变体与推理」，用引擎与单例承担「真相与结算」。**

## 2. 三层架构（心智模型）

1. **传统层（Deterministic Substrate）**  
   时间、位置、背包、作物、挖矿钓鱼结算、`GameManager` 权威状态、**存档结构**。  
   **原则**：模型与 prompt **不得**直接改写这些状态；只能通过**已定义、可校验的 API**。

2. **AI 层（LLM & Tools）**  
   对话文案、任务创意、叙事摘要、NPC 语气与细节。  
   **原则**：输出必须是**约定形状**（JSON / 枚举 / 固定段落），经校验后再进入游戏状态。

3. **Agentic 层（期望建设，影响面大）**  
   **有状态的多步决策与编排**：谁在什么时机规划、谁执行、谁写回世界、失败如何重试或降级。  
   **原则**：可观测（日志/trace）、可关断（纯本地规则 fallback）、**不复制**一套与 `GameManager` 冲突的平行经济或任务真相。

## 3. 三条硬契约（防严重偏离）

在任意大功能合并前，用这三条自问；不满足则先补设计再写代码。

### 3.1 确定性边界

**问**：哪些字段永远必须由规则与存档权威持有？（例：金币、物品 ID 与数量、任务完成布尔、房屋等级。）

**要求**：这些字段的变更路径可枚举；LLM 输出只能作为**建议**，经规则层**采纳/拒绝**后再写入。

### 3.2 合法形状（Schema / 契约）

**问**：模型返回的 JSON 或文本，字段集、类型、最大长度是否固定？谁在何处校验？

**要求**：客户端或服务端至少一处 **parse + 校验**；非法则 **明确降级**（模板句、idle、跳过本次生成），**静默吞错**视为技术债。

### 3.3 失败与降级

**问**：超时、空响应、格式错误、玩家离线时，玩家看到什么？存档是否仍一致？

**要求**：每条对外 LLM 路径都有 **fallback**；关键路径禁止「无响应则卡死」。

## 4. Agentic 层：建设时额外遵守的约定

- **单一写入口**：世界状态变更尽量经同一层（如 `GameManager` + 明确方法），避免多个 autoload 各写各的。
- **任务/目标对象结构化**：Agent 产出应是 `action` + `parameters` 一类，与后端已有 `AgentEngine` 决策 JSON 思路对齐；客户端若扩展，保持**同一语义**。
- **可调试**：关键决策打日志（npc_id、摘要、是否采用 fallback），便于判断是 prompt 问题还是规则问题。
- **与玩法里程碑对齐**：Agentic 能力应在 `TODO.md` 阶段 B（或单独「Agentic 里程碑」）上有**可验收条目**，避免无限堆 autoload 却无闭环。

---

## 5. 反模式（视为架构风险）

- 同一业务规则在 **GDScript、prompt、JSON 表** 三处各写一份，且无单一权威。
- 为赶进度把 **未校验的模型输出** 直接写入存档或任务状态。
- 新增 **第三个**「AI 管理器」而不合并职责（已有 `AIAgentManager`、`AdvancedAIManager` 等，合并与边界应被持续审视）。
- 用大量 **UI/手感打磨** 替代 **AI 闭环与校验** 的排期（前者要做，但不能掩盖后者缺口）。

## 6. 下一步计划（建议顺序，可拆进里程碑）

1. **一纸契约**：列出「确定性字段清单」+「LLM 允许写入的 API 列表」（哪怕初版很短）。  
2. **选一条链路做示范**：例如 NPC 对话或每日叙事，端到端：**请求 → 响应 schema → 校验 → 写回/展示 → 日志**。

## 8. 当前偏离程度评估（截至文档编写时，定性）

本评估基于仓库现状：**玩法与数据驱动内容持续充实**，**AI 基础设施多点多线**，**阶段 B（AI 任务落地、经济闭环、叙事可见化）在路线图中仍大量未完成**。以下为**主观但可复核**的结论。

| 维度 | 判断 | 说明 |
|------|------|------|
| **传统可玩层** | **偏离度低（健康）** | 钓鱼/挖矿/烹饪/农场等多用 `data/*.json` 与系统脚本，符合「单处权威」方向。 |
| **AI 运行时路径** | **偏离度中** | 存在多处 LLM 相关入口（如 `AIAgentManager`、`AdvancedAIManager` 等），若长期不**划界或收敛**，易增认知负担与重复范式。 |
| **服务端 / AgentEngine** | **偏离度低–中（偏健康）** | Python 侧已有路由、决策 JSON 解析与失败回退等模式，与本文「契约+降级」一致，可继续作为**参考实现**。 |
| **Agentic 闭环与阶段 B** | **偏离度中高（主要缺口）** | `TODO.md` 中 B1–B3 仍大量未勾；**「活」的长期来源**若依赖 Agentic 与经济/任务闭环，则当前整体仍偏**传统玩法 + AI 插件**，尚未形成**端到端新范式闭环**。 |
| **表现与迭代（如 A3）** | **偏离度低（非负面）** | UI/季节色等不替代 Agentic，但若**长期占满排期**而 B 段不动，则与「AI 驱动差异化」愿景的**相对偏离**会拉大。 |

**一句话**：**底座与内容管线没有严重失控，但「Agentic + 阶段 B」仍是愿景与现状之间的主要落差；** 风险不在于代码行数，而在于 **未收敛的 AI 入口 + 未闭合的 AI 经济与任务契约**。
