# 配置化开发原则（Agentic 迭代）

本项目后续迭代默认遵循“**先配置，后代码**”的落地顺序：先定义可调参数与边界，再在脚本中读取并执行。

## 1) 单一配置源

- 同一玩法闭环（任务阈值、奖励、经济脉冲、失败策略）应集中在一个配置文件内。
- 示例：`data/quests/managed_chain.json` 统一管理委托链节奏、奖励与市场脉冲。

## 2) 默认兜底

- 脚本读取配置必须有 fallback 默认值。
- 配置缺失/解析失败时，系统可继续运行，且在日志中给出告警。

## 3) 参数边界

- 对配置值做最小/最大约束（如时间阈值、倍率、奖励非负）。
- 避免“错误配置直接破坏玩法”。

## 4) 可观测反馈

- 关键分支（成功、临近超时、失败、AI 回退）都需要玩家可见反馈：
  - `WorldEventFeed`
  - `QuickTip`
  - `QuestLog` 状态标签

## 5) 模板化扩展

- 新增任务链优先走模板配置（例如 `data/quests/chain_templates.json`），而不是复制粘贴脚本分支。
- 代码负责解释模板、推进状态、派发反馈。

## 6) 轻测试闭环

- 每次新增配置能力至少补 1 个最小单测（配置读取、状态推进或分支行为）。
- 测试优先覆盖“最容易回归的规则边界”。

## 7) 文档同步

- 每轮完成后在 `docs/MANAGED_ITERATION_CHECKLIST.md` 记录：
  - 新增配置项
  - 影响的系统
  - 玩家可见变化
