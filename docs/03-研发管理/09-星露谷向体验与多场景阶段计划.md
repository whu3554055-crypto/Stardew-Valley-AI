# 星露谷向体验与多场景大地图 — 阶段改进方案 & TODO List

> **定位**：本文档为**现阶段（当前迭代）**的约定工作内容：在 Godot 4.x 上持续逼近《星露谷物语》式的**大地图 + 多区域场景 + 农场循环**，同时保留本项目已有长板（战斗、任务链、Agentic、存档安全等）。  
> **执行方式**：按下方 **TODO List 自上而下、逐项勾选实现**；每完成一项建议单独提交 Git，便于回滚与评审。

---

## 1. 结论摘要（Godot 能否做到星露谷）

- **可以。** 星露谷的成功主要来自**内容量、美术统一性、系统设计**，而非「只有某引擎才能实现」。
- Godot 在 **2D TileMap、UI、跨平台、数据驱动、Autoload 全局状态** 上与需求高度匹配。
- 本项目已在 **Godot 4.6** 中具备：农场逻辑、矿洞与战斗、任务与存档签名、环境音与部分 HUD。**多场景骨架与 §4 清单项已闭环**；持续迭代重点在 **统一像素美术管线、作物/NPC 数据化与内容厚度、大规模地块性能与存档结构的深化**（与 §2 各维度一致）。

## 2. 改进方案总览（分维度）

### 2.1 世界与地图（对标：海边、森林、山洞、农场、镇子）

| 维度 | 星露谷参考 | 本项目方向 |
|------|------------|------------|
| 空间结构 | 多区域、大地图、区域间过渡自然 | **分场景（推荐）**：镇子 / 农场 / 森林 / 海边 / 矿洞（可多层）等独立 `tscn`，用**传送点 + 出生点**串联；长期可加「大地图选点」或公交式过渡 |
| 单场景规模 | 单区可很大 | 每区 TileMap **可扩展**（如 80×50 级→更大），用 **Chunk/区域边界** 预留性能优化 |
| 视觉层次 | 多层瓦片、遮挡、室内外 | **TileMapLayer**（或等价多层）+ **Y-Sort**（玩家与树、建筑前后关系） |
| 导航 | 玩家心理地图清晰 | **统一小地图或区域名 HUD**（已有活动区文案，可升级为区域标题 + 可选地图） |

### 2.2 系统与数据（对标：种田、社交、时间）

- **时间/季节**：继续强化与 **作物表、商店库存、NPC 日程** 的绑定（数据驱动）。
- **作物与工具**：生长、浇水、季节限制、肥料等尽量 **JSON/Resource 单表**，任务系统用现有 `track_event` 对齐。
- **NPC**：最小闭环为 **日程占位 + 对话资源 + 好感数值**；AI/记忆在之上增强，而非替代底层规则。
- **战斗与经济**：保留差异化；与多场景衔接（仅矿洞/特定区域触发深度战斗亦可）。

### 2.3 美术与体验（对标：像素统一、「像游戏」）

- **像素规范**：固定 tile 尺寸（如 16 或 32）、主色板、NEAREST 采样统一。
- **占位策略**：先统一 **一套主 tileset** 覆盖草地/路/水岸，再逐区替换；多边形逻辑区可保留为碰撞/触发，**逐步减少「字当 UI」**。
- **UI**：右侧日志 Tab、矿区折叠等继续迭代；世界内**长说明改为靠近触发**。

### 2.4 技术与风险（对标：存档、规模、Mod）

- **存档**：维持版本号 + 校验；多场景后存档需 **当前场景名 + 位置 + 各场景持久化状态**（或统一 WorldState 字典）。
- **性能**：高密度作物避免「一作物一 Node」；采用 **网格数据 + 批量绘制/TileMap 表现** 的路线预研。
- **Mod（可选）**：优先 **JSON 覆盖数据路径**（`user://mods`），再考虑脚本类 Mod。

## 3. 多场景大地图 — 推荐架构（供实现对照）

以下作为 **TODO 的技术约定**，实现时可微调命名，但建议保持语义一致。

1. **场景清单（示例）**  
   - `world_town` — 镇子、商店、NPC 聚集  
   - `world_farm` — 玩家农场、建筑升级区  
   - `world_forest` — 伐木、采集、通往他区  
   - `world_beach` — 钓鱼、沙滩采集  
   - `world_mine` — 矿洞入口层 + 深层可子场景或同场景多层 Y 带（与现有矿带逻辑衔接）

2. **过渡机制**  
   - `Area2D` 或数据驱动的 **传送组件**：`target_scene`、`target_spawn_id`。  
   - **Autoload**（如扩展 `GameManager` 或新建 `WorldRouter`）：`change_world(scene_path, spawn_id)`，淡入淡出可选。

3. **出生点**  
   - 每场景 `Markers/Spawn_*` 或 `Node2D` 命名约定，避免硬编码坐标散落。

4. **相机与玩家**  
   - 玩家可置于 **高一层级**由当前场景加载，或每场景实例化玩家并在 `change_world` 时迁移（二选一，TODO 中先定一种）。

5. **与现有单一大场景的关系**  
   - **迁移策略**：当前 `main.tscn` 可逐步拆为「枢纽」或「农场主场景」，逻辑区（厨房、矿点等）随区域迁到对应 `world_*`。
